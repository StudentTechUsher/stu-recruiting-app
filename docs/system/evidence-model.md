# Evidence Model Contract

## Purpose
Define the shared artifact and evidence structure used by candidate and recruiter workflows, including role capability model context and coaching-ready evidence semantics.

## Terminology
| Context | Term | Definition |
| --- | --- | --- |
| User-facing | Evidence Profile | Candidate-facing representation of capabilities and linked evidence. |
| User-facing | Role capability model | Company-role-specific target model used to interpret evidence readiness. |
| Internal | Artifact | Typed evidence record with provenance and verification metadata. |
| Internal | Artifact version | Source-specific artifact version preserved for auditability. |

## Artifact Record Contract
| Field | Requirement |
| --- | --- |
| `artifact_id` | Required unique identifier. |
| `profile_id` | Required owner profile reference. |
| `artifact_type` | Required typed value (`coursework`, `project`, `internship`, `employment`, `certification`, `leadership`, `club`, `competition`, `research`, `test`). |
| `artifact_data` | Required normalized payload with core fields plus typed fields. |
| `file_refs` | Optional provenance and file references. |
| `created_at` | Required timestamp. |
| `updated_at` | Required timestamp. |

## Evidence Linkage Contract
| Field | Requirement |
| --- | --- |
| `capability_profile_id` | Required when linkage is evaluated in a selected target context. Nullable only for baseline onboarding views. |
| `capability_id` | Required capability reference. |
| `artifact_ids` | Required supporting artifact ID list. |
| `verification_breakdown` | Required counts by `verified`, `pending`, and `unverified`. |
| `verification_tier_breakdown` | Required counts by tier (`self_asserted`, `platform_backed`, `verified`). |
| `evidence_confidence` | Required evidence confidence class (`strong`, `moderate`, `limited`, `insufficient`). |
| `last_evidence_at` | Required most recent supporting evidence timestamp. |

## Evidence Confidence Semantics
| Confidence class | Meaning | Minimum expectation |
| --- | --- | --- |
| `strong` | Capability claim supported by multiple consistent and credible evidence records. | At least one verified or platform-backed record and no major contradiction. |
| `moderate` | Capability claim supported, but breadth or verification depth is limited. | At least one relevant record with partial verification support. |
| `limited` | Capability claim exists but evidence is sparse, weak, or mostly self-asserted. | May include only self-asserted evidence or weak provenance. |
| `insufficient` | Not enough evidence to support capability claim. | No credible linked evidence. |

## Verification Tier Semantics
| Tier | Meaning | Typical source examples |
| --- | --- | --- |
| `self_asserted` | Candidate-entered or extracted without independent source validation. | Manual entry, resume extraction, unvalidated claim |
| `platform_backed` | Backed by platform-origin data with source provenance but not full independent verification. | GitHub metadata, Kaggle profile data, LeetCode public profile data |
| `verified` | Completed accepted verification path. | Transcript parse, validated document or attestation path |

## Coaching Linkage Contract
| Field | Requirement |
| --- | --- |
| `coaching_action_id` | Optional action identifier generated by Capability Fit Coaching. |
| `expected_evidence` | Optional structured expectation describing evidence artifact that would strengthen credibility. |
| `expected_evidence.artifact_type` | Required when `expected_evidence` exists. |
| `expected_evidence.verification_target` | Optional target verification tier for credibility improvement. |
| `expected_evidence.acceptance_hint` | Optional practical completion hint for candidate UX. |

## Version Retention and Presentation Rules
| Rule ID | Rule |
| --- | --- |
| EV-001 | Keep all artifact versions as provenance-linked records. |
| EV-002 | Never delete historical versions as part of merge or derivation. |
| EV-003 | Define a primary display artifact for UI only. |
| EV-004 | Primary display priority uses verification strength, then completeness, then recency. |
| EV-005 | UI display priority must not imply data replacement or data loss. |

## Deterministic Merge and Display Decision Table
| Condition | Primary display behavior | Persistence behavior |
| --- | --- | --- |
| Higher verification strength exists | Display highest verification representation | Keep all versions |
| Verification tie, completeness differs | Display most complete representation | Keep all versions |
| Verification and completeness tie | Display most recent representation | Keep all versions |
| Identity ownership conflict | Candidate identity remains authoritative | Keep all versions |

## Weak and Unverifiable Evidence Rules
| Rule ID | Rule |
| --- | --- |
| EV-WEAK-001 | Weak or unverifiable evidence remains visible with explicit trust labeling. |
| EV-WEAK-002 | Weak evidence may support exploratory coaching suggestions but cannot be framed as strong readiness proof. |
| EV-WEAK-003 | Decision-Ready Candidate Package must clearly distinguish weak evidence from verified evidence. |

## Explicit Exclusions
| Exclusion | Rule |
| --- | --- |
| Ranking outputs | Evidence linkage must not produce candidate ranking signals. |
| Score-centric UX dependency | User-facing summaries must not depend on opaque composite scores. |
| Hidden transform divergence | Recruiter-visible evidence relationships must match candidate-visible relationships. |
